# 生成AIでプレフィックスを一括変更する

## 概要

RINKの運用が進み、知識体系が大きく変わる場合、**生成AI（ChatGPT、Gemini等）を活用して、プレフィックスを一括変更する**ことが可能。この仕組みにより、大規模な構造変更を数分で実行できる。

## なぜ必要か

### 成長段階での構造見直し

RINKが成熟すると、初期設計の限界が見えてくる。

```
初期：
t.1.a.1 スケール基礎
t.2.a.1 コード基礎
t.3.a.1 ハーモニー基礎

学習進行後：
t.0.a.1 基礎中の基礎（新たに必要）
t.1.a.1 スケール基礎
t.2.a.1 コード基礎（拡張）
t.3.a.1 ハーモニー基礎
t.4.a.1 応用テクニック（新規追加）
```

このとき、全プレフィックスを手動で変更するのは非常に手間。

## 生成AIによる一括変更

### 指示例

```
ChatGPTへの依頼：

次のリストを処理してください。

元のプレフィックス形式: t.1 → t.2 に変更するプログラム（Pythonまたはシェルスクリプト）を作成して。

条件：
- "t.1" で始まるプレフィックスを "t.2" に置換
- ファイル内の全ての "t.1.a.1", "t.1.a.2" など、関連するプレフィックスも含める
- 誤り検出機能を含める

例：
t.1.a.1 → t.2.a.1
t.1.a.2 → t.2.a.2
...
```

### 所要時間
- 生成AIがコード生成：約2～5分
- 実装・テスト：5～10分
- 実行：数秒～数分

合計：15分以下で、数百項目のプレフィックス一括変更が可能。

## 大規模変更の例

### パターン1：上位カテゴリの見直し
```
変更前：
g. = Guitar
t. = Theory
p. = Plugin

変更後（楽器別に再編成）：
🎸.t = Guitar Theory
🎸.t. → gt.に変更
...
```

### パターン2：階層の深さ調整
```
変更前：
t.4.a.1

変更後（さらに細分化）：
t.4.a.1.i (さらに深い階層)
```

## 実装のコツ

### 1. ルール整備
- 変更前のプレフィックス体系を文書化
- 変更後のルールを明確に定義
- エッジケース（例外）を列挙

### 2. テスト環境での実行
- 本体のRINKではなく、テスト用ワークスペースで試行
- 動作確認後、本体に適用

### 3. バックアップ
- 変更前にVaultをバックアップ
- 何か問題があった場合の復旧手段を確保

## RINK運用での意義

### 柔軟性の実現
「いつでもやり方を変えられる」という思想が、実際に機能する。

従来は、一度決めたプレフィックス体系は固定的。
- 手動変更は膨大な手間
- 結果として、ルール変更を躊躇
- システムが硬直化

RINKでは、プログラムを活用することで、この問題を解決。

### スケーラビリティ
数百項目を超える規模になっても、構造の大幅な見直しが可能。

成長に応じた進化を、躊躇なく実行できる。

## 重要な注意事項

### AIを信用しすぎない
- 生成AI自体は完璧ではない
- 小規模なテストケースで検証必須
-複雑な論理は、複数パターンで確認

### ドキュメント化
- 変更内容を記録
- 「なぜ変更したのか」を明記
- 将来の参考に

この仕組みこそが、RINKが「死なないシステム」であり、「生涯の相棒」として機能する理由。
