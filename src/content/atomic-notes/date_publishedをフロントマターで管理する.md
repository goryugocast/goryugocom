# date_publishedをフロントマターで管理する

## 概要
従来はファイル名に日付を含めていたが、`date_published`プロパティをフロントマターで管理することで、ファイル名をシンプルに保ちながら日付情報を活用できる。

## 従来の問題点

### ファイル名に日付を含める課題
```
20251113_セミナーリマインド.md
20251120_セミナー告知.md
```

- ファイル名が長くなる
- ファイル数が多いと日付でのスクロールが必要
- 実際には日付で検索することが少ない
- AI指示時に日付更新の手間

## 新しい方法：フロントマター管理

### ファイル構造
```markdown
---
date_published: 2025-11-13T00:00:00.000Z
topic: seminar040
---

# セミナーリマインド
```

### メリット

1. **ファイル名がシンプル**
   ```
   ks.rem.040_🔔セミナー直前リマインド.md
   ```
   日付を含めないので、意味情報に注力できる

2. **Baseで日付検索が可能**
   ```
   date_published > 2025-01-01
   ```
   フロントマターのプロパティなら、Baseクエリで日付検索ができる

3. **日付更新が簡単**
   Baseの動的な日付計算機能を使えば、自動的に古いノートと新しいノートが分離される

4. **複製時の日付更新**
   テンプレートではなく複製を前提にすれば、日付の自動更新も可能

## 実装例：デイリーノートでの活用

```yaml
---
date_published: 2025-01-21
type: daily
---

# 2025年1月21日
```

Baseで`date_published >= Today`のように動的にフィルタリングすれば、「今日以降のタスク」「明日のスケジュール」などを自動表示。

## AI時代のメリット

- プレフィックスだけでAIに指示可能
- 日付情報はプロパティで一元管理
- 複数の日付属性（作成日、公開日、期限など）を同時に管理可能

## プロパティの種類分け

```yaml
date_created: 2025-01-15      # 作成日
date_published: 2025-01-20    # 公開日
date_deadline: 2025-02-01     # 期限
date_modified: 2025-01-21     # 最終編集日
```

複数の日付属性を区別して管理することで、より柔軟な情報管理が可能。
