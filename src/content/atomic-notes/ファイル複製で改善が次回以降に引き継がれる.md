# ファイル複製で改善が次回以降に引き継がれる

## 概要
ファイルの複製を前提とすることで、毎回の改善が自動的に次のサイクルに引き継がれ、システム全体が螺旋的に洗練されていく。

## 改善の循環

### テンプレート方式での問題
```
テンプレート（A）
    ↓
新規作成（テンプレートA→B）
    ↓
改善（B は改善されるが A には影響なし）
```
新しく作成されるファイルはテンプレートのままで、改善が活かされない。

### 複製方式での改善循環
```
ファイルA（改善前）
    ↓
複製→ファイルB（Aの改善点を継承）
    ↓
さらに改善（Bに新しい工夫を追加）
    ↓
複製→ファイルC（A + B の全改善を継承）
    ↓
さらに改善...
```
毎回、これまでの改善をすべて含んだ状態から開始できる。

## 実装例：セミナー運用

### セミナー039→040の改善例

**セミナー039での工夫：**
- Baseクエリで関連ファイルを自動集約
- topic: seminar039 で一元管理
- プロパティ設定を整理

**セミナー040（複製直後）：**
すべての工夫が自動的に継承される。

**セミナー040での新しい改善：**
- アーカイブ配信セクションを追加
- メタデータ構造を調整
- Base クエリを最適化

**セミナー041（複製直後）：**
セミナー039 + セミナー040 の全改善を継承した状態でスタート。

## 複製時の手順

### ステップ1：ファイル複製
```
セミナー040 を右クリック
→ Duplicate current file
```

### ステップ2：最小限の修正
```yaml
# 変更する部分
- topic: seminar040 → seminar041
- date_published: 2025-10-13 → 2025-11-13
- 1回目固有の記述だけ修正

# 変更しない部分
- Base クエリの設定
- プロパティ構成
- セクション構成
- リンク構造
```

### ステップ3：新しい改善を追加
セミナー041を実施する中で、さらに改善点が見つかったら追加。その改善がセミナー042に引き継がれる。

## 改善の蓄積メカニズム

### 1ヶ月ごとのセミナー運用

```
9月（セミナー037）
 ↓
10月（セミナー038 = 037 + 1ヶ月分の改善）
 ↓
11月（セミナー039 = 037 + 038 + 1ヶ月分の改善）
 ↓
12月（セミナー040 = 037 + 038 + 039 + 1ヶ月分の改善）
```

3個先のセミナーになると、3ヶ月分の改善が蓄積されている状態。

## 複製の威力

### 見える改善：ファイル開く度に実感
- 「あ、こんなセクションができていたのか」
- 「プロパティ設計が洗練されている」
- 「Baseクエリがこんなに優れていたのか」

### 目に見えない改善：システムの洗練
- ナレッジ管理の効率が向上
- 関連情報へのアクセス時間が短縮
- エラーが減る

## デメリットと対策

### デメリット：ファイル数が増える
```
セミナー037, 038, 039, 040... と増え続ける
```

### 対策：Baseで整理
```yaml
# Base クエリで当月のセミナーだけを表示
topic: "seminar%"
date_published >= 2025-11-01
```

整理・検索はBaseに任せて、ファイル自体は保存し続ける。

## 重要な原則

複製による改善の循環は、「毎回、前回より良い状態でスタートできる」という強力なメカニズム。テンプレート方式では達成できない。
