## 概要

ノートのフロントマター（YAML形式のメタデータ）に`project:`フィールドを追加することで、プロジェクトベースの検索・フィルタリングが可能になります。

## フロントマターの基本構造

```yaml
---
title: ノートのタイトル
tags:
  - タグ1
  - タグ2
project: music
---
```

## プロジェクト属性の利点

### 検索性の向上
- `project: music`で全ノート抽出
- Baseプラグインでの条件検索
- CLI検索での高速フィルタ

### 情報の階層化
- タグ：低粒度（複数付与可能）
- プロジェクト：中粒度（単一指定）
- フォルダ：高粒度（ファイルシステム）

### メタデータの一元管理
- すべての情報をYAMLに集約
- 外部ツールでの利用も容易
- 他のシステムへの移行も簡単

## 運用ルール

### プロジェクト名の決定

#### パターン1：分野ベース
```
project: music
project: photography
project: writing
```

#### パターン2：仕事案件ベース
```
project: clientA
project: clientB
project: book2024
```

#### パターン3：混合型
```
project: music      # 趣味
project: work_proj1 # 仕事
```

### 命名規則の統一
- 小文字で統一
- ハイフン使用（スペース避ける）
- 3-15字程度に制限

## Baseプラグインとの連携

### クエリの書き方
```
project = "music"
```

### 複雑な条件検索
```
where project = "music" and tags contains "機材"
```

## フロントマター管理の実践

### 新規ノート作成時
- プロジェクトが決まっている場合は必ず指定
- テンプレートで自動挿入するのが効率的

### 既存ノートの整理
- 一括編集で統一
- VS Codeなどのテキストエディタで検索置換

### プロジェクト完了時
- プロジェクトフィールドを削除
- または`completed: true`を追加

## 注意点

### Publishでの制約
- Publishでは`project:`フィールドはほぼ表示されない
- Publish用には別途手動整理が必要
- Baseクエリ結果もPublishに反映されない

### DataViewの代替
- DataViewも同様の機能だが、Publishで表示されない
- Baseプラグインも同じ制約
- 人に見せるページは手動で構成

## ベストプラクティス

### 少数のプロジェクトに絞る
- 5個以上のプロジェクトは管理困難
- 「その他」カテゴリーの活用
- 定期的なプロジェクト統廃合

### 定期的な見直し
- 月単位での一覧確認
- 完了プロジェクトの処理
- 新規プロジェクト追加の是非判断

### 他のメタデータとの組み合わせ
```yaml
project: music
status: active
priority: high
tags:
  - 機材
  - レビュー
```

メタデータ戦略次第で、Obsidianの検索効率は劇的に向上します。
