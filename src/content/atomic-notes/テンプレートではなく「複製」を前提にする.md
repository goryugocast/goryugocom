# テンプレートではなく「複製」を前提にする

## 概要
Obsidianの効果的な運用では、テンプレート機能ではなく、既存ファイルの複製（デュプリケート）を前提にすることで、改善が自動的に次回以降に引き継がれる。

## 従来の方法：テンプレート

### テンプレートの問題点

```
template.md
↓
新規ノート作成（テンプレートから生成）
↓
テンプレートを改善してもこのノートには反映されない
```

テンプレートを改善しても、既に作成されたノートには影響しない。

### テンプレートの改善は「点」
- テンプレート改善後のノートには適用される
- テンプレート改善前のノートは変わらない
- システム全体が進化しない

## 新しい方法：複製を前提

### 複製ワークフロー

```
セミナー039（完成度の高い状態）
↓
複製してセミナー040を作成
↓
セミナー039の改善点が自動的にセミナー040に引き継がれる
```

### 複製の利点

1. **改善が自動的に継承**
   - セミナー039で導入した新しいプロパティ設計が039にも反映
   - 新しいセクション構成が自動で引き継がれる

2. **段階的な進化**
   - 毎回のセミナーで微調整
   - その改善が次のセミナーに引き継がれる
   - システム全体が螺旋的に洗練される

3. **前回のベストプラクティスを活用**
   - テンプレートより最新の状態を反映
   - 「今の最高状態」を次のファイルの基本にできる

## 実装例

### セミナー管理での活用

```yaml
# セミナー039（改善後）
---
topic: seminar039
date_published: 2025-10-01
status: completed
notes: "今回から新しいセクション構成を導入"
---

↓ Duplicate → セミナー040

# セミナー040（複製直後）
---
topic: seminar040
date_published: 2025-11-01
status: in_progress
notes: "セミナー039の改善されたセクション構成を継承"
---
```

topic番号を040に変更し、date_publishedを更新するだけで、その他のプロパティ設定やセクション構成がすべて引き継がれる。

## テンプレートとの併用

### ハイブリッドアプローチ

```
初期テンプレート（基本構造）
↓
複製で改善を積み重ねる
↓
相当進化した時点でテンプレート更新
```

新規プロジェクトでは基本テンプレートから開始し、軌道に乗ったら複製に切り替える。

## プロ仕様のポイント

### 複製時のチェックリスト
```
[ ] topic / project 情報を新しい番号に変更
[ ] date_published を新しい日付に変更
[ ] 1回目の固有情報だけを修正
[ ] その他の設定・セクション構成はそのまま
```

### 改善の例

```
複製1回目（セミナー039→040）
- プロパティ追加：urgency（緊急度）

複製2回目（セミナー040→041）
- セクション構成を調整
- 新しいメタデータを追加

複製3回目（セミナー041→042）
- Base クエリを最適化
```

毎回の改善が蓄積される。

## デイリーノートへの応用

```
デイリーノート（昨日）
↓
複製 → 今日のデイリーノート
↓
昨日の完了したタスク、メモ、リンク構造がすべて継承される
```

毎日改善される「今の最高状態」のデイリーノートで1日を始められる。
