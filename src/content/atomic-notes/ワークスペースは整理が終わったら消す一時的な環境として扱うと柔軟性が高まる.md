# ワークスペースは整理が終わったら消す一時的な環境として扱うと柔軟性が高まる

## 基本概念

RINKの最大の特徴の1つが、**一時的な専用環境の作成と削除が可能**なこと。「永続的な構造」ではなく、「その時々の必要に応じて存在する環境」として、ワークスペースを捉える。

## ワークスペースのライフサイクル

### 1. 作成
特定の目的のために、新しいワークスペースを作成。

例：
- 「Ableton Liveをマスターしよう」→ **Live専用ワークスペース**作成
- 「クライアントA向けの提案書を作成する」→ **ClientA専用ワークスペース**作成

### 2. 運用
その目的に集中して、ワークスペース内でノートを作成・整理。

### 3. 達成
学習/作業が完了し、知識が統合される。

### 4. 削除
目的を達成したワークスペースは**削除**。

## なぜ削除するのか

### ワークスペース数の爆増を防ぐ
- 複数のワークスペースを並行運用すると、UI上で見にくくなる
- 「あ、あのワークスペースどれだっけ？」という混乱を防止

### 一時的な集中環境
- クライアント案件ごと、学習テーマごとに一時的なモードを作成
- 完了したら削除し、次の目的にスペースを使う

### 清潔性の保持
ワークスペース一覧に、「不要な過去の環境」が溜まらない。

## 削除後のデータ

### 重要な注意事項
- ワークスペースを削除しても、**ノート（.mdファイル）は削除されない**
- 削除されるのは、「ワークスペース設定」のみ

### 再利用の可能性
```
example.md
ファイル自体は vault に存在し続ける
↓
別のワークスペースや、統合後のRINKで再度アクセス可能
↓
「Ableton Live学習」の完了後、
その知識を「🎵 Music Theory」の本体に統合
```

## 実践的な活用例

### 時間軸での活用

#### 月初
```
今月の目標：シンセサイザーをマスター
→ Synth専用ワークスペース作成
```

#### 月中
```
Synthの学習が進行
ワークスペース内で関連ノートが成長
```

#### 月末
```
シンセの基礎知識が完成
本体の「🎵 Music Theory」に統合
→ Synth専用ワークスペース削除
```

### 季節ごとの活用
- 「春：ギター技術習得」→ 春終了時に削除
- 「夏：DAW操作マスター」→ 夏終了時に削除
- 「秋：作曲理論習得」→ 秋終了時に削除

## RINK運用での位置づけ

### 柔軟性の具現化
「いつでもやり方を変えられる」という思想の最たる例。

- 新しい学習テーマが出たら、すぐに専用ワークスペースを作成
- 達成したら、スパッと削除
- クラッターが溜まらず、常にシンプルな環境を保持

### スケーラビリティ
100個のワークスペースを作って、終わったら削除—という自由度があることで、Obsidianの運用がより**柔軟で、実験的**になる。

ワークスペースは「永遠の構造」ではなく、「その時々の相棒」という位置づけ。
