# 現実的に処理できる量のノートだけをレビュー対象にする

## 概要
GTDやSpaced Repetitionのシステムを導入する際、全てのノートをレビュー対象にするのではなく、「実際に処理可能な量」に限定することで、システムの継続性と実用性を確保する戦略。

## 「全部レビュー」の罠

### メンタルモデルの問題
- 「完璧なシステムを作ろう」という心理
- 理想的には「全てのノートをレビュー対象に」
- ただし、実際には処理不可能

### 数値での実感
```
例：レビュー対象ノート 300 個
1日のレビュー能力：3～5 個
必要期間：300 ÷ 3 = 100日（3ヶ月以上）
```

- 3ヶ月以内に全て見直す？ → 不可能
- 見直しが遅延 → システムの信頼性が低下
- 最終的に「見直さない」状態に

## セレクティブなレビュー対象の決定

### `#review`タグの活用
- 「本当に重要なノート」だけに `#review` タグを付与
- Spaced Repetition のレビュー対象を自動的に制限
- 現実的に処理可能な量に留める

### 対象の目安
```
全ノート数：500個
レビュー対象：50～100個（10～20%）
1日のレビュー数：3～5個
全体見直し期間：15～35日（約1ヶ月）
```

これなら月単位で 1 周できる。

## 実装方法

### セレクティブな基準の設定

#### 対象にする
- 「今後、複数回見返す可能性がある」ノート
- 学習系・知識系のノート
- シリーズものの記録ノート
- プロジェクト関連のノート

#### 対象にしない
- アーカイブ済み（完了した）タスク
- 一度限りの情報（特定の日の打ち合わせ記録など）
- 参考資料的なノート（辞書的な役割）
- 雑メモ

### タグの付与時期

#### 作成時に付与
```
ノートリファクターで新規作成した時点で、
自動的に #review タグを付与
```

#### 後から付与
```
週次レビュー時に「今後見返す価値がある」
と判断したノートに追加
```

## レビュー対象の例

### 対象になるノート
- [[Obsidianのプラグイン活用法]]
- [[音楽理論：ハーフディミニッシュコード]]
- [[執筆プロセスの最適化]]
- [[プロジェクトX：進捗管理]]

### 対象にならないノート
- 「2025-08-21 営業会議」（特定日の記録）
- 「参考：ChatGPT 出力結果」（一度限りの参考資料）
- 「アーカイブ：250819 セミナー配信」（完了済み）

## メリット

### システムの継続性
- 現実的な量だけを見直し対象に
- 完了サイクルが短い（1ヶ月程度）
- 「見直せなかった」という罪悪感が減る

### 自動的な優先順位付け
- `#review` が付いたノートは「自分が大事だと判定した」ノート
- 自然と重要度の高いノートだけが残る

### 関心の深さの可視化
- 1ヶ月に複数回レビューされるノートは「本当に大事」
- Spaced Repetition の見直し頻度で優先順位が明確化される

## 注意点

### タグの削除も重要
- 「もう不要」と判断したら、#review タグを削除
- または、アーカイブフォルダに移動
- レビュー対象を動的に管理することが継続のコツ

### 「完璧さ」との決別
- 「本当は全て見直したい」という気持ちは自然
- ただし、その気持ちが「システムの破綻」を招く
- 「80%の精度で 100% 継続する」方が価値が高い

### 定期的な見直し
- 3ヶ月ごとに、#review タグ対象を再評価
- 不必要になったノートのタグを削除
- 新しく重要になったノートを追加

## 心理的なフレーム

### パラダイムシフト
```
従来：「完璧なシステムを作ろう」
  → 理想の 50% も実装できず、放棄

新しい考え：「現実的に継続できるシステムを作ろう」
  → 80% の精度で 100% 継続
```

### できることとできないことの線引き
- 1日 5 個は見直せる？ → YES
- 50 個は見直せる？ → NO
- なら、1日 5 個のシステムを作る

## 応用例

### フェーズ別の管理
```
フェーズ 1（最初の 1ヶ月）：#review ノート 20 個
フェーズ 2（2ヶ月目）：#review ノート 50 個
フェーズ 3（3ヶ月目以降）：#review ノート 100 個
```

段階的に増やしていくことで、無理なく拡大できる。
